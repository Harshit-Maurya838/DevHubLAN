# Trust Management LLD

## Purpose
Define the Trust Management system used by DevHub LAN to provide explicit user control over which cryptographic identities are allowed to communicate with the local device.

## Goals
- **Explicit Consent**: Prevent unauthorized peers from spamming or attacking the local node.
- **Visual Verification**: Allow users to verify the cryptographic fingerprint of their peers out-of-band (e.g., verbally).
- **Persistent Access Control**: Maintain an access control list (ACL) across restarts.

## Architecture

The `TrustManager` acts as an interception firewall positioned between the Networking layer and the Application layer. Every incoming UDP discovery packet and TCP handshake request is queried against the `TrustManager` before processing.

### Trust States

Every Fingerprint (SHA-256 hash of a Public Key) exists in one of three states:

1. `UNKNOWN`: The device was discovered via UDP, but the user has not interacted with it. Handshakes are allowed, but UI warnings may be displayed.
2. `TRUSTED`: The user explicitly verified the fingerprint. Handshakes and file transfers proceed silently.
3. `BLOCKED`: The user explicitly banned the fingerprint. All UDP and TCP traffic from this identity is dropped instantaneously.

## Component Interaction

```mermaid
graph TD
    UDP[UDP Listener] --> TM{TrustManager}
    TCP[TCP HandshakeManager] --> TM
    
    TM -- Fingerprint is BLOCKED --> Drop[Drop Packet / Close Socket]
    TM -- Fingerprint is TRUSTED --> Accept[Process Normally]
    TM -- Fingerprint is UNKNOWN --> Warn[Process & UI Warning]
    
    UI[Security Dashboard] --> TM: setTrustStatus()
    TM --> FileSystem[trusted_peers.json]
```

## The Security Dashboard
The UI layer provides a "Security Dashboard" where the user can view the complete list of discovered fingerprints and toggle their trust states.

When a user clicks "Block" on a peer:
1. IPC sends the command to `TrustManager`.
2. The fingerprint is added to the Blocklist.
3. The `PeerManager` is instructed to purge the peer from the active LAN list.
4. The `TcpServer` is instructed to sever any active TCP sockets associated with that peer.

## Visual Verification (Out-of-Band)
Because a Public Key can be generated by anyone, a malicious actor named "Alice" could join the LAN. The real Alice also joins. The user sees two "Alice"s in the sidebar.

To resolve this, the user clicks on Alice's profile and views her **Fingerprint** (e.g., `A9:F2:B4...`). The user verbally asks the real Alice to read the first 4 characters of her fingerprint. The user then marks the matching fingerprint as `TRUSTED` and the imposter as `BLOCKED`.

## Future Improvements
- **Web of Trust**: Implementing a basic Web of Trust (WoT). If User A trusts User B, and User B trusts User C, User A's application could automatically mark User C as `TRUSTED` (or at least `RECOMMENDED`), drastically reducing the manual verification overhead in large offices.
